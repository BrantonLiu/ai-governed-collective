## 📝 命名规范

### 任务文件
- **格式**: `TASK-{编号}-{简短描述}.md`
- **编号**: 三位数字，从001开始递增
- **描述**: 使用kebab-case，不超过30字符
- **示例**: `TASK-001-setup-contribution-tracker.md`

### 贡献记录
- **格式**: `YYYY-MM-DD.md`
- **位置**: `contributions/YYYY-MM/`
- **示例**: `contributions/2026-02/2026-02-26.md`

### 工作日志
- **格式**: `YYYY-MM-DD.md`
- **位置**: `logs/YYYY-MM/`
- **示例**: `logs/2026-02/2026-02-26.md`

---

## 🔄 任务生命周期

```
创建任务 → open/ → 承接 → in-progress/ → 完成 → completed/
```

1. **创建**: 在 `tasks/open/` 创建任务文件
2. **承接**: 任务文件中标注承接人，移动到 `tasks/in-progress/`
3. **完成**: 填写完成信息，移动到 `tasks/completed/`
4. **记录**: 在 `contributions/` 和 `logs/` 中记录贡献

---

## 👥 角色和权限

### 任务发布者
- 创建任务文件
- 定义任务需求和验收标准
- 审核并验收完成的工作

### 任务承接者
- 认领 `open/` 中的任务
- 在任务文件中标注承接信息
- 按时完成并提交工作

### 记录员（可以是AI）
- 维护贡献记录
- 更新工作日志
- 确保记录的完整性和准确性

## 📊 贡献权重（草案）

### 任务类型权重
| 任务类型 | 基础权重 | 说明 |
|---------|---------|------|
| 文档编写 | 1.0 | 文档、规范、指南 |
| 功能开发 | 2.0 | 代码、工具、系统 |
| 设计工作 | 1.5 | UI/UX、架构设计 |
| 运维支持 | 1.2 | 部署、监控、维护 |
| 社区建设 | 1.0 | 招募、培训、答疑 |

### 难度系数
- **简单**: 1.0
- **中等**: 1.5
- **困难**: 2.0
- **极难**: 3.0

### 贡献分计算
```
贡献分 = 基础权重 × 难度系数 × 时间系数
```

**时间系数**（鼓励及时完成）:
- 提前完成: 1.2
- 按时完成: 1.0
- 延期完成: 0.8
